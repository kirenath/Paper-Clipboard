Build a lightweight private clipboard web app.

The app is a personal temporary text transfer tool, not an automatic clipboard sync tool. It is used to manually paste text on one device and copy it from another device. It will eventually be deployed on a VPS behind Cloudflare Tunnel and Cloudflare Access. The app itself must also have a single shared password login.

Use:

- Next.js App Router
- TypeScript
- Tailwind CSS
- shadcn/ui components
- Responsive design
- Light theme by default
- Dark mode toggle
- Paper / note-card visual style

The UI language should be Simplified Chinese.

Do not implement:

- automatic clipboard monitoring
- file upload
- multi-user account system
- automatic expiration
- offline cache
- realtime sync
- end-to-end encryption
- Supabase
- Neon
- external database SDKs

For v0 preview, use a temporary in-memory data store, but structure the code so the data store can later be replaced by SQLite easily.

---

## Product goal

Create a private clipboard manager for storing short or medium text snippets such as:

- VPN links
- temporary email passwords
- one-time codes
- config snippets
- server commands
- device-to-device temporary text

The app should feel lightweight, fast, private, and personal.

## Authentication requirements

Implement single shared password authentication.

Important security requirements:

- Do not check the password only on the client.
- Do not expose the app password in client-side code.
- The password must be read from server-side environment variable:

```env
APP_PASSWORD=change-me
SESSION_SECRET=change-me-to-a-long-random-string
```

- Login page posts password to a server API route.
- Server validates the password.
- On successful login, set an HTTP-only session cookie.
- Cookie should use:
  - httpOnly
  - sameSite=lax
  - secure in production
  - reasonable max age, for example 7 days
- All app pages and API routes should require a valid session.
- Provide logout button that clears the session cookie.

For v0 preview/development, if environment variables are missing, allow a clearly marked development fallback password:

```text
dev-password
```

But keep a comment in code saying this fallback must not be used in production.

Required auth routes:

```text
POST /api/auth/login
POST /api/auth/logout
GET /api/auth/session
```

## Data model

Use this schema as the single source of truth for the future SQLite database.

Create a file:

```text
db/schema.sql
```

With this SQLite-compatible schema:

```sql
CREATE TABLE IF NOT EXISTS clipboard_groups (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,
  sort_order INTEGER,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS clipboard_tags (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,
  sort_order INTEGER,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS clipboard_items (
  id TEXT PRIMARY KEY,
  group_id TEXT,
  title TEXT,
  content TEXT NOT NULL,
  sort_order INTEGER,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (group_id) REFERENCES clipboard_groups(id) ON DELETE SET NULL
);

CREATE TABLE IF NOT EXISTS clipboard_item_tags (
  item_id TEXT NOT NULL,
  tag_id TEXT NOT NULL,
  PRIMARY KEY (item_id, tag_id),
  FOREIGN KEY (item_id) REFERENCES clipboard_items(id) ON DELETE CASCADE,
  FOREIGN KEY (tag_id) REFERENCES clipboard_tags(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_clipboard_items_group_id
ON clipboard_items (group_id);

CREATE INDEX IF NOT EXISTS idx_clipboard_items_sort_order
ON clipboard_items (sort_order);

CREATE INDEX IF NOT EXISTS idx_clipboard_items_updated_at
ON clipboard_items (updated_at DESC);

CREATE INDEX IF NOT EXISTS idx_clipboard_groups_sort_order
ON clipboard_groups (sort_order);

CREATE INDEX IF NOT EXISTS idx_clipboard_tags_sort_order
ON clipboard_tags (sort_order);

CREATE INDEX IF NOT EXISTS idx_clipboard_item_tags_tag_id
ON clipboard_item_tags (tag_id);
```

Important model rules:

- `clipboard_items` is the main table.
- One item can belong to zero or one group.
- If `group_id` is null, show it as group:

```text
默认
```

- One item can have zero or more tags.
- Tags are many-to-many through `clipboard_item_tags`.
- `title` is optional.
- `content` is required.
- `sort_order` is optional.
- Smaller `sort_order` means higher priority.
- If `sort_order` is null, the item is a normal item.
- `created_at` and `updated_at` should be ISO strings generated by the application.
- `id` should be generated by the application using `crypto.randomUUID()`.

## Data access architecture

Create a clearly separated data access layer:

```text
lib/clipboard-store.ts
```

All CRUD operations must go through this file.

Do not let React components directly access the database or mock data.

Do not scatter data logic across UI components.

For v0 preview, implement `lib/clipboard-store.ts` with an in-memory store and some seed data.

But write it as if it can later be replaced by SQLite without changing the UI or API routes.

The store should expose functions similar to:

```ts
listClipboardItems(options?: {
  groupId?: string | null
  tagIds?: string[]
  query?: string
}): Promise<ClipboardItem[]>

createClipboardItem(input: {
  title?: string | null
  content: string
  groupId?: string | null
  tagIds?: string[]
  sortOrder?: number | null
}): Promise<ClipboardItem>

updateClipboardItem(id: string, input: {
  title?: string | null
  content?: string
  groupId?: string | null
  tagIds?: string[]
  sortOrder?: number | null
}): Promise<ClipboardItem>

deleteClipboardItem(id: string): Promise<void>

listClipboardGroups(): Promise<ClipboardGroup[]>

createClipboardGroup(input: {
  name: string
  sortOrder?: number | null
}): Promise<ClipboardGroup>

updateClipboardGroup(id: string, input: {
  name?: string
  sortOrder?: number | null
}): Promise<ClipboardGroup>

deleteClipboardGroup(id: string): Promise<void>

listClipboardTags(): Promise<ClipboardTag[]>

createClipboardTag(input: {
  name: string
  sortOrder?: number | null
}): Promise<ClipboardTag>

updateClipboardTag(id: string, input: {
  name?: string
  sortOrder?: number | null
}): Promise<ClipboardTag>

deleteClipboardTag(id: string): Promise<void>
```

Deletion rules:

- Deleting an item should remove that item and its tag relations.
- Deleting a group should not delete items. Items in that group should become default group, meaning `groupId = null`.
- Deleting a tag should not delete items. It should only remove tag relations.

## API routes

Create API routes that call the store functions.

Required routes:

```text
GET    /api/items
POST   /api/items
PATCH  /api/items/[id]
DELETE /api/items/[id]

GET    /api/groups
POST   /api/groups
PATCH  /api/groups/[id]
DELETE /api/groups/[id]

GET    /api/tags
POST   /api/tags
PATCH  /api/tags/[id]
DELETE /api/tags/[id]
```

All API routes must check session authentication.

Use JSON request and response bodies.

Validation requirements:

- Empty `content` is not allowed for clipboard items.
- Empty group name is not allowed.
- Empty tag name is not allowed.
- Trim titles, group names, and tag names.
- Empty title should be stored as null.
- Empty sort order should be stored as null.
- Duplicate group or tag names should show a friendly error.
- Return proper HTTP status codes.

## Sorting rules

### Clipboard item sorting

Default item order:

1. Items with `sortOrder !== null` come first.
2. Smaller `sortOrder` comes first.
3. If two items have the same `sortOrder`, newer `updatedAt` comes first.
4. Items with `sortOrder === null` come after manually sorted items.
5. Normal items are sorted by newer `updatedAt` first.

Equivalent concept:

```ts
manual sorted items first,
then sortOrder ascending,
then updatedAt descending,
then unsorted items by updatedAt descending
```

### Group sorting

Groups should be sorted by:

1. Items with `sortOrder !== null` first.
2. Smaller `sortOrder` first.
3. Then name ascending.

The default group is virtual and not stored in the database.

Display the default group as:

```text
默认
```

The default group should appear near the top of the group list.

### Tag sorting

Tags should be sorted by:

1. Tags with `sortOrder !== null` first.
2. Smaller `sortOrder` first.
3. Then name ascending.

## Important implementation constraints

- Keep data access isolated in `lib/clipboard-store.ts`.
- Do not directly access mock arrays inside React components.
- Do not implement database-specific logic outside the store layer.
- Do not expose server secrets to the client.
- Do not use browser localStorage as the primary persistence layer.
- Do not over-engineer the app.
- Prefer readable, maintainable code.
- Prefer simple API routes and clear component boundaries.
- Use optimistic UI only if it does not make the code too complex.
- Use toast notifications for actions.
- Make sure the app works well on mobile.
